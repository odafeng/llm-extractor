# 1. 地端優先、雲端 fallback 的 LLM 架構

Status: Accepted (2026-06-16)

## Context

病理報告含病患個資 (PHI)，醫院資安政策通常**禁止將病患資料送往外部雲端 API**。
但純地端的開源模型在萃取品質與速度上未必足夠，且不是每個執行環境都有 GPU。
我們需要一個既能滿足「資料不出院」、又能在需要時借助前沿雲端模型的架構。

## Decision

採用**地端優先、雲端 fallback** 的雙軌設計：

- 預設使用地端 Ollama（`qwen2.5:14b`）進行推論。
- 地端呼叫失敗時，自動 fallback 到雲端 GPT（需設定 `OPENAI_API_KEY`）。
- 提供 `USE_CLOUD_ONLY=1` 旗標，可在有授權的情境下強制只用雲端。
- 地端與雲端共用同一個 OpenAI-compatible client 介面與同一份 prompt。

## Consequences

**正面**
- PHI 預設不離開院內環境；可離線運作；地端推論無 API 成本。
- 地端／雲端共用程式碼與 prompt，維護成本低、行為一致。
- 單一旗標即可切換到純雲端，方便在去識別化資料上做品質比較。

**負面 / 取捨**
- 地端模型的萃取品質可能低於前沿雲端模型，需以驗證資料確認可用性。
- **fallback 是隱性的**：地端故障時 PHI 會在使用者未必察覺的情況下送往雲端，
  屬於資安風險，應加上明確的政策、紀錄與（必要時）使用者確認。
- 需維運地端 Ollama（模型下載、GPU/記憶體資源）。
- 兩條路徑的輸出可能有細微差異，跨模型結果不完全可重現。
