# 2. 去識別化與分離保存的再識別金鑰

Status: Accepted (2026-06-16)

## Context

原始病理報告的 JSON 含病歷號 (chart number) 等可識別個資。
研究與 LLM 萃取流程需要在去識別化資料上進行，但仍需保留「能回溯到原始病患」的能力，
以便日後串接臨床結果。如何同時兼顧可分享性與可回溯性是關鍵。

## Decision

由 `deidentification.py` 執行去識別化：

- 移除 JSON 中的 `chart_number`，改以流水號 `P001`、`P002` … 命名與標記。
- 另存一份對照金鑰 `id_mapping_key.csv`（`Study_ID` ↔ 原始病歷號／檔名／路徑）。
- 對照金鑰與含病歷號的 cohort 檔一律 **gitignore，只保留在本機**，永不進版控。
- 去識別化後的報告（`for_study_deidentified/`）才可進版控／分享。

## Consequences

**正面**
- 去識別化資料可安全地進版控、分享與供 LLM 處理。
- 金鑰與資料分離，降低單點外洩即可再識別的風險。
- 流水號保留了可回溯性，便於日後串接臨床資料。

**負面 / 取捨**
- 對照金鑰一旦外洩即可完全再識別，必須嚴格保管。
  （本專案確實曾發生含病歷號的 cohort 檔誤入公開 repo 的事件，已用 `git filter-repo`
  清除歷史並導入 gitleaks 防護，詳見 commit 記錄。）
- 目前僅移除**結構化的病歷號**；報告自由文字中若殘留姓名、日期等，未必被清除，
  需另做文字級去識別化（尚未實作）。
- 流水號依檔案掃描順序產生，若重跑時來源順序不同會造成 ID 對應錯位。
